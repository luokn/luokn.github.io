内核杂谈（一）：Linux 里是怎么切换地址空间的？

2026年6月20日 · 阅读需 19 分钟
Kun Luo
Software Engineer
提示

CPU 切任务：Task A → Task B，不只是换寄存器/栈/调度信息，还得换 CR3、页表、TLB 上下文。不然 B 会看见 A 的地址空间。

switch_mm 负责把 CPU 从旧地址空间切到新地址空间。本文先回顾 PCID/Local ASID 机制，再聊 Linux 6.15 左右进主线的 Global ASID。

（文中代码基于 Linux 7.1 源码整理，相关机制在 6.15 已合入主线。）

先复习一下 x86 的 TLB 与 PCID​

  • 每个逻辑 CPU 都有独立的 CR3、CR4 以及 TLB 状态。所有 CPU 共享同一套物理内存，但各自维护自己的地址转换缓存。

  • 因为页表根目录（PGD）按 4KB 对齐，CR3 的低 12 位原本全是 0。x86 开启 PCID（CR4.PCIDE=1）后，就把这 12 位复用成 PCID 字段。

  • TLB 里存的条目不是简单的 VA → PA，而是 (PCID, VA) → PA。不同地址空间即使拥有相同的虚拟地址，只要 PCID 不同，就可以同时驻留在 TLB 里不冲突。

  • 写 CR3 是一条序列化指令，能够保证旧地址空间下的内存访问完成，之后再开始基于新 CR3 的地址翻译。也正因为这个特性，switch_mm 被内核当成一个重要的内存屏障点。

  • 在 PCID 机制下，写 CR3 时可以设置 NOFLUSH 位（bit 63），保留对应 PCID 下已有的 TLB 缓存。切换地址空间时，CPU 不必每次都清空 TLB，而是可以在多个地址空间之间复用之前的条目。

  • 对于跨 CPU 的 TLB shootdown，需要发 IPI （Inter-Processor Interrupt）通知远端 CPU 刷掉对应 PCID 的条目，这依赖操作系统提供的中断处理机制。除此之外，AMD 还为 x86 提供了 INVLPGB（broadcast TLB invalidation）指令，可以直接失效指定 PCID 下的所有条目（所有 CPU），省掉软件发 IPI 的开销。不过这个指令是 AMD Zen3 首次引入的，Intel 还不支持。

补充说明
  • 现在的 CPU 已经有 5 级页表支持，VA 也干到了 57 位（12 + 5 × 9）。不过这不影响本文要讲的切换逻辑，下面默认按 4 级页表理解。
  • Intel 的 LAM（Linear Address Masking）会把 CR3 里没用到的位掩码掉，给软件当元数据用。默认关闭，本文不展开。
  • Linux 6.15 左右进主线的 "Global ASID"，就是用来配合这类硬件广播失效的。不过 x86 的 PCID 仍然只有 4096 个（PTI 开启时折半），所以内核采用 Local ASID 为主、符合条件再迁移到 Global ASID 的策略。

一些关键的数据结构​

mm_struct​

这个不用细说，mm_struct 是描述一个进程地址空间最核心的结构。这里只看跟切换相关的成员：

struct mm_struct {
    pgd_t *pgd;

    /* 架构相关的上下文 */
    mm_context_t context;

    /*
     * mm_cpumask 需要放在 mm_struct 末尾，
     * 因为它的大小随 nr_cpu_ids 动态变化。
     */
    char flexible_array[] __aligned(__alignof__(unsigned long));
};
  • pgd：指向当前 mm 的页表根目录（PGD）。切换地址空间时，最终会把它的物理地址写进 CR3。
  • context：架构相关的上下文，我们目前关注 x86 架构，x86 下主要包含 Unique ID、TLB Generation、Global ASID 等关键字段。
  • 末尾的 flexible_array 通过 mm_cpumask(mm) 访问，是一个 cpumask_t，用来记录哪些 CPU 正在使用这个 mm，这个字段相当重要。TLB shootdown 就是靠它知道该通知谁。

mm_context_t​

x86 的 mm_context_t 定义在 arch/x86/include/asm/mmu.h，跟切换相关的字段如下：

typedef struct {
    /*
     * ctx_id 唯一标识一个 mm_struct，不会复用，0 也不是有效值。
     */
    u64 ctx_id;

    /*
     * 任何需要刷这个 mm 的 TLB 的代码，都会先改页表，
     * 再递增 tlb_gen，最后执行 flush。底层通过比较 tlb_gen
     * 判断当前 CPU 缓存是否过期。
     */
    atomic64_t tlb_gen;

    /* 以下两项只在配置 BROADCAST_TLB_FLUSH 时存在 */

    /*
     * 非零表示这个进程在所有 CPU 上使用同一个 ASID，
     * 从而可以利用硬件远程 TLB 失效，比如 AMD INVLPGB。
     */
    u16 global_asid;

    /* 正在从 Local ASID 迁移到 Global ASID 的过渡标志 */
    bool asid_transition;
} mm_context_t;
  • ctx_id：唯一的 mm 标识，ASID 复用时靠它判断 TLB 条目是否还属于同一个 mm。
  • tlb_gen：每做一次页表修改并准备 flush，这个值就加 1。CPU 上记录的 tlb_gen 如果落后，说明本地 TLB 有 stale 条目。
  • global_asid / asid_transition：Global ASID 相关，后面专门讲。

tlb_state​

tlb_state 是每个 CPU 私有的状态，记录当前加载的是哪个 mm、ASID 是什么、各个 ASID 槽位对应的 tlb_gen 等：

struct tlb_state {
    /*
     * 当本地中断开启时，loaded_mm 应该和 CR3 保持一致。
     * 在 lazy TLB 模式下，loaded_mm 仍指向之前的用户 mm，
     * 但 CR3 可能已经切回 init_mm。
     */
    struct mm_struct *loaded_mm;

#define LOADED_MM_SWITCHING ((struct mm_struct *)1UL)

    /* 最后一次运行的用户 mm，用于 IBPB 优化 */
    union {
        struct mm_struct *last_user_mm;
        unsigned long   last_user_mm_spec;
    };

    u16 loaded_mm_asid;    /* 当前 CPU 正在使用的 ASID */
    u16 next_asid;         /* 下一个可分配的动态 ASID */

    /*
     * 当页表改变方式需要失效所有上下文时置位，
     * 下次切上下文时会清掉非当前 ctxs[] 对应的缓存。
     */
    bool invalidate_other;

#ifdef CONFIG_ADDRESS_MASKING
    u8 lam;
#endif

    /* PTI 下记录哪些用户 PCID 下次切回用户态时需要 flush */
    unsigned short user_pcid_flush_mask;

    unsigned long cr4;

    /*
     * 每个 ASID 槽位保存它对应的 mm ctx_id 和 tlb_gen。
     * ASID 0..5 称为 Dynamic ASID，共 TLB_NR_DYN_ASIDS=6 个。
     */
    struct tlb_context ctxs[TLB_NR_DYN_ASIDS];
};
DECLARE_PER_CPU_ALIGNED(struct tlb_state, cpu_tlbstate);

struct tlb_context {
    u64 ctx_id;
    u64 tlb_gen;
};
注意

  • tlb_state 是本地信息，而 mm_context_t 是全局信息。切换时就是通过比较这两边的 tlb_gen 来决定要不要 flush。

  • 这里的TLB_NR_DYN_ASIDS在目前版本是 6，意味着每个 CPU 本地最多只能缓存 6 个不同 mm 的 ASID 条目。超过这个数量的进程要么在切换时走慢路径频繁 flush TLB，要么迁移到 Global ASID（如果满足条件）。至于为啥是 6，我觉得可能是因为：

    1. 这个值如果太大的话的，每个 CPU 上缓存的 mm 数量变多，相应的每个 mm 的 mm_cpumask 里 CPU 数量变多，TLB shootdown 需要发送 IPI 的 CPU 数量也变多了。
    2. 每次切换 mm 时，需要遍历 ctxs[] 来找 ctx_id 匹配的槽位，如果 TLB_NR_DYN_ASIDS 太大了，这个线性也会变慢。
    3. 这个值如果太小的话，频繁切换的进程可能经常走慢路径 flush TLB。
    4. sizeof(struct tlb_state) 可能原本是想凑两个 cache line（128B）的大小，不过现在struct tlb_state已经超过 128B 了。

tlb_state_shared​

struct tlb_state_shared {
    bool is_lazy;
};
DECLARE_PER_CPU_SHARED_ALIGNED(struct tlb_state_shared, cpu_tlbstate_shared);

is_lazy 之所以放在独立的 shared per-cpu 变量里，是因为其他 CPU 在进行 TLB shootdown 时需要读到它。tlb_state 是 cache-line 对齐的 per-cpu 数据，访问路径更私有；把 is_lazy 单独拎出来，让跨 CPU 读取的开销更可控。

任务切换流程：schedule() → switch_mm_irqs_off​

调度器在进行任务切换（schedule()）时，会走到 context_switch()，地址空间切换就是在这里发生的：

schedule()
    |
    +- context_switch()
        |
        ?- 如果 next 是内核线程，即：next->mm == NULL
        |       |
        |       +- enter_lazy_tlb(prev->active_mm, next)
        |
        ?- 如果 next 是用户进程，即：next->mm != NULL
        |       |
        |       +- membarrier_switch_mm(...)
        |       |
        |       +- switch_mm_irqs_off(prev->active_mm, next->mm, next)
        |
        +-- switch_to()      // 切寄存器、栈

context_switch直接调用 switch_mm_irqs_off()，这个函数名强调了irqs_off，也就是切换地址空间时必须关中断，switch_mm() 这个包装函数也是一进去就关中断：

void switch_mm(struct mm_struct *prev, struct mm_struct *next,
               struct task_struct *tsk)
{
    unsigned long flags;

    local_irq_save(flags);
    switch_mm_irqs_off(NULL, next, tsk);
    local_irq_restore(flags);
}

switch_mm_irqs_off() 这个函数一进来就用 WARN_ON_ONCE(!irqs_disabled()) 断言中断必须是关的：

void switch_mm_irqs_off(struct mm_struct *unused, struct mm_struct *next,
                        struct task_struct *tsk)
{
    ...
    /* We don't want flush_tlb_func() to run concurrently with us. */
    if (IS_ENABLED(CONFIG_PROVE_LOCKING))
        WARN_ON_ONCE(!irqs_disabled());
    ...
}

为什么 switch_mm 必须关中断？​

原因很简单：在切换的那一小段时间里，cpu_tlbstate.loaded_mm、CR3、mm_cpumask() 三者必须保持一致。如果这时候来了中断，尤其是 TLB shootdown IPI，观察到的状态可能就是中间态。

具体来说，x86 的 flush_tlb_func() 本身也是中断上下文里跑的（它靠 IPI 触发），它一上来会读 cpu_tlbstate.loaded_mm 和 cpu_tlbstate_shared.is_lazy。如果 switch_mm 执行到一半、IRQ 突然打开，IPI 处理器看到的 loaded_mm 和实际 CR3 不一致，就可能做出错误判断：

  • 以为某个 mm 还在运行，结果没刷；
  • 以为已经切走了，结果刷了不该刷的；
  • 更麻烦的是 nmi_uaccess_okay()：NMI 处理程序要通过 loaded_mm 判断能不能安全访问用户空间，如果 loaded_mm 和 CR3 对不上，就可能访问到错误的地址空间。

所以代码在真正写 CR3 之前，先把 loaded_mm 设成一个特殊哨兵值 LOADED_MM_SWITCHING（这个值其实就是 ((struct mm_struct *)1UL)，不指向任何合法的 mm_struct）：

/*
 * Indicate that CR3 is about to change. nmi_uaccess_okay()
 * and others are sensitive to the window where mm_cpumask(),
 * CR3 and cpu_tlbstate.loaded_mm are not all in sync.
 */
this_cpu_write(cpu_tlbstate.loaded_mm, LOADED_MM_SWITCHING);

这段窗口期必须原子的，办法就是整个 switch_mm_irqs_off() 都关中断执行。等 CR3 写完了、loaded_mm 和 loaded_mm_asid 都更新完毕，再开中断。

另外，写 CR3 本身还自带全内存屏障 + 核心序列化效果，membarrier() 系统调用依赖这一点来保证 rq->curr 更新和返回用户态之间的顺序。所以 switch_mm 不仅是地址空间切换点，也是内核里的一个关键同步点。

快路径：prev == next​

如果 cpu_tlbstate.loaded_mm == next，说明这个 CPU 已经加载了目标 mm，不一定需要重新写 CR3。

if (prev == next) {
    /*
     * 如果当前 mm 正在往 Global ASID 迁移，
     * 即使 prev == next 也要把 CR3 换成 Global ASID。
     */
    if (mm_needs_global_asid(next, prev_asid)) {
        next_tlb_gen = atomic64_read(&next->context.tlb_gen);
        ns = choose_new_asid(next, next_tlb_gen);
        goto reload_tlb;
    }

    /*
     * 已经在用 Global ASID：硬件广播失效会保证所有 CPU 的 TLB
     * 都是最新的，直接返回。
     */
    if (is_global_asid(prev_asid))
        return;

    /*
     * 非 lazy 模式：同一个进程内的线程切换，不需要 flush。
     */
    if (!was_lazy)
        return;

    /*
     * 之前是 lazy 模式，需要检查 tlb_gen 判断是否需要补刷。
     */
    smp_mb();
    next_tlb_gen = atomic64_read(&next->context.tlb_gen);
    if (this_cpu_read(cpu_tlbstate.ctxs[prev_asid].tlb_gen) ==
            next_tlb_gen)
        return;

    /* lazy 期间页表被改过，fall through 去慢路径补刷 */
    ns.asid = prev_asid;
    ns.need_flush = true;
}

慢路径：prev != next（真正切换 mm 和 flush TLB）​

当 prev != next 时，执行真正的切换：

} else {
    /* 进程间切换时可能需要 IBPB/L1D flush 等缓解措施 */
    cond_mitigation(tsk);

    /*
     * 标记 CR3 即将改变。nmi_uaccess_okay() 等代码对
     * mm_cpumask()、CR3、loaded_mm 不一致的窗口很敏感。
     */
    this_cpu_write(cpu_tlbstate.loaded_mm, LOADED_MM_SWITCHING);

    /*
     * 确保本 CPU 在 next 的 mm_cpumask 里，这样后续的 TLB shootdown
     * 才能通知到我们。cpumask_set_cpu() 的原子写自带 smp_mb()。
     */
    if (next != &init_mm && !cpumask_test_cpu(cpu, mm_cpumask(next)))
        cpumask_set_cpu(cpu, mm_cpumask(next));
    else
        smp_mb();

    next_tlb_gen = atomic64_read(&next->context.tlb_gen);
    ns = choose_new_asid(next, next_tlb_gen);
}

然后统一走到 reload_tlb:：

reload_tlb:
new_lam = mm_lam_cr3_mask(next);
if (ns.need_flush) {
    VM_WARN_ON_ONCE(is_global_asid(ns.asid));
    this_cpu_write(cpu_tlbstate.ctxs[ns.asid].ctx_id, next->context.ctx_id);
    this_cpu_write(cpu_tlbstate.ctxs[ns.asid].tlb_gen, next_tlb_gen);
    load_new_mm_cr3(next->pgd, ns.asid, new_lam, true);
} else {
    /* 这个 ASID 已经是最新的，写 CR3 时带 NOFLUSH */
    load_new_mm_cr3(next->pgd, ns.asid, new_lam, false);
}

/* 确保 CR3 写完后才更新 loaded_mm */
barrier();

this_cpu_write(cpu_tlbstate.loaded_mm, next);
this_cpu_write(cpu_tlbstate.loaded_mm_asid, ns.asid);
cpu_tlbstate_update_lam(new_lam, mm_untag_mask(next));

if (next != prev) {
    cr4_update_pce_mm(next);
    switch_ldt(prev, next);
}

注意 ns.need_flush 对 Global ASID 来说永远是 false，因为 Global ASID 由硬件广播失效保证一致性，不需要在写 CR3 时清 TLB。

ASID 与 PCID 的对应关系​

Linux 内部把 "ASID" 和 "PCID" 这两个名字区分开，避免搞混：

/*
 * ASID - [0, TLB_NR_DYN_ASIDS-1]
 *        每个 CPU 本地动态分配的 mm 标识。
 *        [TLB_NR_DYN_ASIDS, MAX_ASID_AVAILABLE-1]
 *        全局 ASID，所有 CPU 上相同。
 *
 * kPCID - [1, MAX_ASID_AVAILABLE]
 *         实际写入 CR3 的 PCID 值，等于 ASID + 1，因为 PCID 0 被保留。
 *
 * uPCID - 如果开启 KPTI，每个 mm 还有对应的用户态 PCID，
 *         通过给 kPCID 置上 PTI_USER_PCID_BIT 得到。
 */

build_cr3 和 build_cr3_noflush 负责拼 CR3 值：

static inline unsigned long build_cr3(pgd_t *pgd, u16 asid, unsigned long lam)
{
    unsigned long cr3 = __sme_pa(pgd) | lam;

    if (static_cpu_has(X86_FEATURE_PCID)) {
        cr3 |= kern_pcid(asid);
    } else {
        VM_WARN_ON_ONCE(asid != 0);
    }

    return cr3;
}

static inline unsigned long build_cr3_noflush(pgd_t *pgd, u16 asid,
                          unsigned long lam)
{
    VM_WARN_ON_ONCE(!boot_cpu_has(X86_FEATURE_PCID));
    return build_cr3(pgd, asid, lam) | CR3_NOFLUSH;
}

kern_pcid() 做 ASID → kPCID 的映射：

static inline u16 kern_pcid(u16 asid)
{
    VM_WARN_ON_ONCE(asid > MAX_ASID_AVAILABLE);

#ifdef CONFIG_MITIGATION_PAGE_TABLE_ISOLATION
    BUILD_BUG_ON(TLB_NR_DYN_ASIDS >= (1 << X86_CR3_PTI_PCID_USER_BIT));
    VM_WARN_ON_ONCE(asid & (1 << X86_CR3_PTI_PCID_USER_BIT));
#endif

    /*
     * 加 1 是为了把 PCID 0 腾出来，留给不懂 PCID 的代码路径使用。
     * 这样万一有谁 save/restore CR3 时把 PCID 丢了，也不会污染 ASID 0 的缓存。
     */
    return asid + 1;
}

所以映射关系是：

概念范围说明
Dynamic ASID0 ~ 5每个 CPU 本地缓存最近用过的几个 mm
Global ASID6 ~ MAX_ASID_AVAILABLE-1所有 CPU 上相同，给多线程进程用
kPCIDASID + 1实际写入 CR3 的 kernel PCID
uPCIDkPCID | PTI bit如果开了 PTI，用户态 PCID 另加一位区分

硬件虽然有 4096 个 PCID（PTI 开启时可用 2048 个），但 Linux 并不会给每个进程一个独立的 ASID。本地只留 6 个 Dynamic ASID 做缓存，再拿一部分高编号 ASID 做 Global ASID；剩下的 ASID 空间并不使用。

警告

如果内核开启 PTI，x86 的 Global ASID/INVLPGB 路径通常会被禁用（6.15 的 stable 修复已经明确 Disable INVLPGB when PTI is enabled）。也就是说，PTI + Global ASID 目前基本互斥。

Lazy TLB机制​

内核线程没有自己的用户地址空间（next->mm == NULL）。调度器在切到内核线程时，不会调用 switch_mm，而是走 enter_lazy_tlb()：

static __always_inline void enter_lazy_tlb(struct mm_struct *mm, struct task_struct *tsk)
{
    if (this_cpu_read(cpu_tlbstate.loaded_mm) == &init_mm)
        return;

    this_cpu_write(cpu_tlbstate_shared.is_lazy, true);
}

它只把 is_lazy 置为 true，不切换 CR3。CPU 继续复用上一个用户进程的页表，这样省了一次 CR3 写和 TLB 刷新。

Process A (mm1)
    ↓
kworker            // 只 set is_lazy，CR3 仍指向 mm1
    ↓
Process A (mm1)    // 切回来时根据 was_lazy 判断要不要补刷

为什么 Lazy CPU 可以不被 shootdown？​

当某个 CPU 处于 lazy 模式时，should_flush_tlb() 会跳过它：

static bool should_flush_tlb(int cpu, void *data)
{
    struct mm_struct *loaded_mm = per_cpu(cpu_tlbstate.loaded_mm, cpu);
    struct flush_tlb_info *info = data;

    /*
     * 按 switch_mm_irqs_off() 里的写顺序，确保读到的 is_lazy
     * 至少和 loaded_mm 一样新。
     */
    smp_rmb();

    /* Lazy TLB 会在下一次上下文切换时自己刷新。 */
    if (per_cpu(cpu_tlbstate_shared.is_lazy, cpu))
        return false;

    /* 没有 mm 表示是内核内存 flush */
    if (!info->mm)
        return true;

    /* 切换窗口期按最坏情况处理 */
    if (loaded_mm == LOADED_MM_SWITCHING)
        return true;

    /* 目标 mm 正在这个 CPU 上跑，且不是 lazy */
    if (loaded_mm == info->mm)
        return true;

    /* 在 cpumask 里但没加载？周期性 trim */
    if (info->trim_cpumask)
        return true;

    return false;
}

跳过的前提是：页表本身不会被释放。如果 info->freed_tables 为真，说明旧页表页可能被回收，lazy CPU speculative 访问到新内容会出乱子，这时必须强制发 IPI：

if (info->freed_tables || mm_in_asid_transition(info->mm))
    on_each_cpu_mask(cpumask, flush_tlb_func, (void *)info, true);
else
    on_each_cpu_cond_mask(should_flush_tlb, flush_tlb_func,
                          (void *)info, 1, cpumask);

从 lazy 切回真实 mm 时的回补​

switch_mm_irqs_off() 一进来就读 was_lazy，如果是 lazy，先清掉标志：

bool was_lazy = this_cpu_read(cpu_tlbstate_shared.is_lazy);
...
if (was_lazy)
    this_cpu_write(cpu_tlbstate_shared.is_lazy, false);

在 prev == next 的快路径里，如果 was_lazy 为真，会比较本地 tlb_gen 和 mm->context.tlb_gen。如果 lazy 期间页表被改过，就 fall through 到慢路径，重新加载 CR3 并刷新 TLB。

leave_mm()​

如果某个代码路径不能容忍 lazy（比如 use_temporary_mm()），可以主动调用 leave_mm()，强制切到 init_mm：

void leave_mm(void)
{
    struct mm_struct *loaded_mm = this_cpu_read(cpu_tlbstate.loaded_mm);

    if (loaded_mm == &init_mm)
        return;

    WARN_ON(!this_cpu_read(cpu_tlbstate_shared.is_lazy));
    switch_mm(NULL, &init_mm, NULL);
}

Global ASID 的渐进式迁移​

Global ASID 的出现是为了解决一个问题：多线程进程在多个 CPU 上同时跑时，每次页表修改都要给其他 CPU 发 IPI 刷 TLB，开销很大。如果能让所有 CPU 上这个进程用同一个 PCID，再配合 INVLPGB 这样的硬件广播失效指令，就可以省掉 IPI。

但 x86_64 的 PCID 数量有限（4096，PTI 下更少），不可能给每个进程都发一个全局 ASID。所以内核采取渐进式迁移：先用 Local ASID，等发现某个进程同时在很多 CPU 上活跃，再给它分配 Global ASID。

什么时候迁移？​

flush_tlb_mm_range() 在走 IPI 路径时，会调用 consider_global_asid(mm)：

static void consider_global_asid(struct mm_struct *mm)
{
    if (!cpu_feature_enabled(X86_FEATURE_INVLPGB))
        return;

    /* 每隔一段时间检查一次，避免每次 flush 都查 */
    if ((current->pid & 0x1f) != (jiffies & 0x1f))
        return;

    /*
     * 如果进程同时在 4 个或更多 CPU 上活跃，
     * 就给它分配 Global ASID。
     */
    if (mm_active_cpus_exceeds(mm, 3))
        use_global_asid(mm);
}

判断标准是 mm_active_cpus_exceeds(mm, 3)：

static bool mm_active_cpus_exceeds(struct mm_struct *mm, int threshold)
{
    int count = 0;
    int cpu;

    /* 快速路径：mm_cpumask 里的 CPU 数不超过阈值，直接返回 */
    if (cpumask_weight(mm_cpumask(mm)) <= threshold)
        return false;

    /* 慢速路径：确认这些 CPU 真的在跑这个进程，而不是 lazy */
    for_each_cpu(cpu, mm_cpumask(mm)) {
        if (per_cpu(cpu_tlbstate.loaded_mm, cpu) != mm)
            continue;
        if (per_cpu(cpu_tlbstate_shared.is_lazy, cpu))
            continue;
        if (++count > threshold)
            return true;
    }
    return false;
}

分配 Global ASID​

static void use_global_asid(struct mm_struct *mm)
{
    u16 asid;

    guard(raw_spinlock_irqsave)(&global_asid_lock);

    /* 已经有 Global ASID 了 */
    if (mm_global_asid(mm))
        return;

    /* Global ASID 池已经空了 */
    if (!global_asid_available) {
        VM_WARN_ONCE(1, "Ran out of global ASIDs\n");
        return;
    }

    asid = allocate_global_asid();
    if (!asid)
        return;

    mm_assign_global_asid(mm, asid);
}

allocate_global_asid() 从 TLB_NR_DYN_ASIDS 往上分配。当快用完时，会调用 reset_global_asid_space()，先用 invlpgb_flush_all_nonglobals() 刷掉所有 Global ASID 条目，再把已释放的 Global ASID 回收复用。

mm_assign_global_asid() 做了件很关键的事：

static inline void mm_assign_global_asid(struct mm_struct *mm, u16 asid)
{
    /*
     * flush_tlb_mm_range() -> broadcast_tlb_flush() ->
     * finish_asid_transition() 必须能在看到 global_asid 之后
     * 立刻看到 asid_transition = true。
     */
    mm->context.asid_transition = true;
    smp_store_release(&mm->context.global_asid, asid);
}

先写 asid_transition = true，再用 release store 写 global_asid。这样任何 CPU 看到 global_asid 非零时，一定也能看到 asid_transition 为真。

过渡期的同步：finish_asid_transition​

迁移不是一瞬间完成的。分配 Global ASID 时，其他 CPU 可能还在用旧的 Local ASID 跑这个进程。必须等所有 CPU 都切到 Global ASID 之后，才能认为迁移完成。

这个工作由 finish_asid_transition() 完成。它在 broadcast_tlb_flush() 调用 INVLPGB 之后执行：

static void finish_asid_transition(struct flush_tlb_info *info)
{
    struct mm_struct *mm = info->mm;
    int bc_asid = mm_global_asid(mm);
    int cpu;

    if (!mm_in_asid_transition(mm))
        return;

    for_each_cpu(cpu, mm_cpumask(mm)) {
        /*
         * 如果远程 CPU 正在 context switch，等它完成，
         * 防止它用旧的 Local ASID 切到目标 mm。
         */
        while (READ_ONCE(per_cpu(cpu_tlbstate.loaded_mm, cpu)) == LOADED_MM_SWITCHING)
            cpu_relax();

        if (READ_ONCE(per_cpu(cpu_tlbstate.loaded_mm, cpu)) != mm)
            continue;

        /*
         * 只要有任何一个 CPU 还没切到 Global ASID，
         * 就发一次 TLB flush IPI。IPI 处理函数里会检查
         * mm_needs_global_asid() 并重新加载 CR3。
         */
        if (READ_ONCE(per_cpu(cpu_tlbstate.loaded_mm_asid, cpu)) != bc_asid) {
            flush_tlb_multi(mm_cpumask(info->mm), info);
            return;
        }
    }

    /* 所有跑这个进程的 CPU 都已经用 Global ASID 了 */
    mm_clear_asid_transition(mm);
}

在过渡期，native_flush_tlb_multi() 会强制给 mm_cpumask 里的所有 CPU 发 IPI（包括 lazy 模式的 CPU），确保它们不会继续持有旧的 Local ASID 条目：

if (info->freed_tables || mm_in_asid_transition(info->mm))
    on_each_cpu_mask(cpumask, flush_tlb_func, (void *)info, true);

迁移完成后的收益​

一旦 asid_transition 被清掉，这个 mm 在所有 CPU 上就都用同一个 PCID 了。以后 flush_tlb_mm_range() 会走 broadcast_tlb_flush()：

static void broadcast_tlb_flush(struct flush_tlb_info *info)
{
    ...
    if (info->end == TLB_FLUSH_ALL) {
        invlpgb_flush_single_pcid_nosync(kern_pcid(asid));
        if (cpu_feature_enabled(X86_FEATURE_PTI))
            invlpgb_flush_single_pcid_nosync(user_pcid(asid));
    } else {
        /* 范围失效 */
        ...
        invlpgb_flush_user_nr_nosync(kern_pcid(asid), addr, nr, pmd);
        ...
    }

    finish_asid_transition(info);

    /* 等待 INVLPGB 广播完成 */
    __tlbsync();
}

这里用 invlpgb_flush_*_nosync() 发起硬件广播，再用 __tlbsync() 等待完成。不需要给每个 CPU 发 IPI，也不需要等待每个 CPU 执行 flush_tlb_func()。对于在大量 CPU 上多线程运行的进程，这部分开销能显著降低。

迁移动作在上下文切换里的体现​

分配 Global ASID 后，当前 CPU 可能下一次 switch_mm_irqs_off() 进来时仍然用着旧的 Local ASID。prev == next 快路径里会检查：

if (mm_needs_global_asid(next, prev_asid)) {
    next_tlb_gen = atomic64_read(&next->context.tlb_gen);
    ns = choose_new_asid(next, next_tlb_gen);
    goto reload_tlb;
}

choose_new_asid() 发现 next->context.global_asid 已经分配好，就直接返回 Global ASID，并且 need_flush = false：

if (cpu_feature_enabled(X86_FEATURE_INVLPGB)) {
    u16 global_asid = mm_global_asid(next);

    if (global_asid) {
        ns.asid = global_asid;
        ns.need_flush = 0;
        return ns;
    }
}

然后 load_new_mm_cr3(..., false) 不带 flush 地重写 CR3，把 PCID 换成 Global ASID。之后这个 CPU 上的 TLB 条目就和其他 CPU 用同一个 PCID 了。

Global ASID 的释放​

进程退出或 exec 时，destroy_context() 会调用 mm_free_global_asid()。被释放的 Global ASID 不会立即放回可用池，而是先标记到 global_asid_freed 位图里，等到 reset_global_asid_space() 时统一用 INVLPGB 刷完再回收。这样避免了一个 CPU 还在用旧 PCID 时另一个 CPU 把同一个 Global ASID 分配出去。

结语​

switch_mm 看起来只是写个 CR3，但里面要处理的事情很多：

  • 利用 PCID 做 TLB 缓存复用；
  • 用 tlb_gen 判断缓存是否过期；
  • 关中断保证切换窗口期的原子性；
  • 对内核线程做 Lazy TLB 优化；
  • 对多线程大进程渐进式迁移到 Global ASID，配合硬件广播失效省掉 IPI。

理解这些细节之后，再看 arch/x86/mm/tlb.c 里的代码就会觉得脉络清晰很多。
